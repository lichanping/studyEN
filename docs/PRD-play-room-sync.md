# PRD: 玩页多人房间同步 MVP

## 背景

当前 [play.html](play.html) 仅在单个浏览器本地内存中维护入座、摇号和消息状态，多个用户同时打开页面时无法互相看到彼此状态，也无法持久化到服务端。

仓库已部署在 Netlify，且已使用 Netlify Functions 与 Netlify Blobs，因此本期 MVP 直接复用 Netlify 资源，不引入额外服务。

## 目标

1. 支持多个移动端/浏览器用户同时进入同一个房间页面。
2. 用户可看到彼此的入座状态、摇号结果与累计摇号总和。
3. 从当前用户进入页面时间开始，仅显示此后产生的系统消息/聊天消息。
4. 修复入座与空座高度不一致导致的 UI 抖动问题。
5. 支持同一用户累计摇号总和统计，并支持清零后重新累计。

## 非目标

1. 不实现 WebSocket / SSE 实时推送，本期采用轮询。
2. 不实现多房间切换，本期固定单房间。
3. 不实现复杂权限系统，仅保留现有“房主位仅甜歌可入座”规则。
4. 不实现长期消息历史查询，用户只看进入页面之后的消息。

## 用户故事

1. 作为进入页面的用户，我可以看到当前已经入座的其他用户。
2. 作为已入座用户，我点击麦序机后，其他用户也能看到我摇到的数字。
3. 作为已入座用户，我多次摇号时，系统会累计显示我的摇号总和。
4. 作为已入座用户，我点击清零后，累计总和从 0 重新开始。
5. 作为新进入页面的用户，我只看到我进入之后产生的消息，但座位状态和累计结果是当前最新状态。

## 功能需求

### P0 多用户同步

1. 页面初始化时，前端向 Netlify Function 拉取当前房间快照。
2. 前端每隔固定时间轮询一次最新房间快照。
3. 入座、离座、摇号、清零、发消息都必须写入服务端。
4. 服务端返回标准房间快照，至少包含：
   - seats：座位占用信息
   - rollTotals：每个用户累计摇号总和
   - seatResults：当前座位最近一次摇号结果
   - messages：消息列表
   - updatedAt：服务端更新时间
5. 新用户进入页面时：
   - seats / seatResults / rollTotals 使用当前服务端快照
   - messages 仅显示 createdAt >= 当前用户 joinAt 的消息

### P0 UI 高度一致

1. 空座和已入座卡片高度必须一致。
2. 昵称区、状态区必须预留固定高度，避免按钮与标签切换后卡片高度变化。
3. 移动端宽度下保持 1 + 4 + 4 的座位布局，不出现跳高和错位。

### P1 摇号累计与清零

1. 每位用户独立维护累计摇号总和。
2. 摇号成功后，累计值 += 本次摇号值。
3. 清零后，该用户累计值重置为 0。
4. 系统消息格式改为：
   - 摇号：`{昵称} 麦序机摇到 {数字}，累计 {总和}`
   - 清零：`{昵称} 已清零`
5. 消息中的数字不再带“号”字。

## 技术方案

### 前端

1. 保留 [play.html](play.html) 页面，改为依赖服务端房间快照渲染。
2. 抽出共享状态模块，供浏览器与 Node 测试复用。
3. 用 localStorage 仅保存当前用户资料与进入时间，不保存房间真状态。
4. 使用轮询刷新房间快照。

### 服务端

1. 新增 Netlify Function：`/.netlify/functions/play-room`。
2. 使用 Netlify Blobs 存储单个房间快照。
3. 支持：
   - GET：获取房间快照
   - POST：提交房间动作（sit / leave / roll / clear / message / profile）
4. 服务端基于当前快照应用动作后再落库，前端不直接上传整份状态。

## 验收标准

1. 两个浏览器同时打开 [play.html](play.html)，A 入座后，B 在轮询周期内能看到 A 已入座。
2. A 摇号后，B 能看到 A 的最新摇号结果和累计值变化。
3. A 连续摇到 1 和 4，累计显示为 5。
4. A 点击清零后，再次摇号从 0 重新累计。
5. 空座与已入座卡片高度一致，无明显跳动。
6. 新用户进入时，只显示其进入之后产生的消息。

## 测试策略

1. 先为共享状态模块添加 Node 测试，覆盖：入座、离座、摇号累计、清零、消息格式。
2. 再实现共享状态模块。
3. 再实现 Netlify Function 与前端接入。
4. 最后运行相关测试并做浏览器手工验证。
